Risk vs. Reward

Speaker Notes - Six Migration Strategies (The 6 Rs)

Introduction:

  • Google Cloud defines six primary migration strategies, often called "the 6 Rs"
  • Each strategy represents a different balance of effort, risk, and potential benefits
  • Most organizations use multiple strategies across their application portfolio

1. Rehost (Lift & Shift) - Green/Low Complexity:

  • Move applications with minimal or no changes
  • Fastest migration approach, lowest risk
  • Best for: Applications that work as-is, tight timelines, limited cloud expertise
  • Limitations: Doesn't leverage cloud-native benefits initially
  • Example: Moving VMs to Google Compute Engine

2. Replatform (Lift & Optimize) - Blue/Medium Complexity:

  • Lift applications and make targeted optimizations
  • Take advantage of some cloud services without major changes
  • Best for: Applications that can benefit from managed services
  • Example: Moving to Cloud SQL instead of self-managed databases

3. Refactor (Move & Improve) - Indigo/Medium-High Complexity:

  • Modify applications to take advantage of cloud capabilities
  • Improve performance, features, cost, and user experience
  • Best for: Applications needing modernization, scalability improvements
  • Example: Containerizing applications for Google Kubernetes Engine

4. Re-architect (Modernize) - Purple/High Complexity:

  • Change how the code functions, not just how it works
  • Break monoliths into microservices
  • Best for: Applications requiring significant scalability, agility improvements
  • Example: Converting monolith to microservices on Cloud Run

5. Rebuild (Remove & Replace) - Red/Very High Complexity:

  • Complete redesign and rewrite as cloud-native application
  • Maximum cloud benefits but highest effort and risk
  • Best for: Legacy applications that can't be modernized, strategic applications
  • Example: Building new serverless application to replace legacy system

6. Repurchase (SaaS Migration) - Yellow/Variable Complexity:

  • Replace with Software-as-a-Service equivalent
  • Often easier than rebuilding, may lose customizations
  • Best for: Common business functions available as SaaS
  • Example: Moving from on-premises email to Google Workspace

Strategy Selection Guidance:

  • Start with rehost for quick wins and learning
  • Use replatform for applications that can easily benefit from managed services
  • Reserve refactor/re-architect for strategic applications
  • Consider repurchase for non-differentiating business functions
  • Most portfolios use 70% rehost/replatform, 30% refactor/rebuild
